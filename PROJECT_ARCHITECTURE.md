# Communication Portal Architecture and Data Flow

This project is a Laravel 12 communication portal built around workspaces, channels, direct messages, tasks, pinned messages, file attachments, notifications, and real-time updates. It uses Laravel Breeze-style authentication, Blade views, Livewire components for interactive screens, Laravel Reverb/Echo for broadcasting, Eloquent models for data access, and Vite/Tailwind/Alpine for frontend assets.

## Root-Level Folders

### `.agents/`

Local agent/workspace metadata. This is not part of the Laravel runtime. It is used by local tooling and should not be treated as application code.

### `.git/`

Git repository metadata. It stores commit history, branches, remotes, and index state. It is not read by the Laravel app at runtime.

### `app/`

The main application code. This is where the business logic lives:

- `app/Enums/` contains backed enums for constrained domain values such as workspace roles and task statuses.
- `app/Events/` contains broadcastable events for real-time features.
- `app/Http/Controllers/` contains request/response controllers for workspaces, channels, DMs, files, tasks, pins, notifications, profiles, users, and authentication.
- `app/Http/Middleware/` contains web middleware, including online-presence tracking.
- `app/Http/Requests/` contains form request validation classes.
- `app/Livewire/` contains interactive UI controllers for chat windows, sidebars, task board, pinned messages, and DMs.
- `app/Models/` contains Eloquent models mapped to the database tables.
- `app/Policies/` contains authorization rules.
- `app/Providers/` contains application service registration.
- `app/Services/` contains service classes, currently including Cloudinary-backed image upload support.
- `app/View/Components/` contains Blade layout component classes.

### `bootstrap/`

Laravel bootstrapping files:

- `bootstrap/app.php` builds the Laravel application, registers routes, adds the `UpdateUserLastSeen` middleware to the web stack, configures trusted proxies, and registers the broadcast channels route file.
- `bootstrap/providers.php` lists service providers.
- `bootstrap/cache/` stores framework-generated cached bootstrap files when optimization commands are run.

### `config/`

Laravel configuration. This folder controls application behavior for auth, broadcasting, cache, database, filesystems, logging, mail, queues, sessions, services, and Reverb.

Important files for this project:

- `config/auth.php` configures authentication providers and guards.
- `config/broadcasting.php` and `config/reverb.php` configure real-time broadcasting.
- `config/database.php` controls the database connection.
- `config/filesystems.php` controls local/public file storage.
- `config/queue.php` controls queued work.
- `config/session.php` controls session handling.

### `database/`

Database schema, factories, and seeders:

- `database/migrations/` creates the schema for users, workspaces, workspace members, workspace join requests, channels, channel users, messages, direct-message conversations, DM participants, direct messages, read states, files, pins, notifications, tasks, jobs, cache, and performance indexes.
- `database/factories/` contains model factories for test/demo data.
- `database/seeders/` contains seeders.

## Core Domain Model

### Users

`App\Models\User` maps to `users` with primary key `user_id`. It uses `password_hash` as the authentication password field and exposes compatibility accessors for `password` and `id`.

A user can:

- belong to many workspaces through `workspace_members`;
- belong to many channels through `channel_user`;
- send channel messages;
- send direct messages;
- create and be assigned tasks;
- receive notifications;
- have online presence tracked in cache.

When a user is deleted, related memberships, channel links, messages, direct messages, created tasks, read states, pins, join requests, and notifications are cleaned up or updated.

### Workspaces

`App\Models\Workspace` maps to `workspace` with primary key `workspace_id`.

A workspace has:

- many channels;
- many workspace-member rows;
- many users through `workspace_members`;
- optional `avatar_url`.

Membership is stored in `workspace_members`, with a role from `App\Enums\WorkspaceRole`.

### Channels

`App\Models\Channel` maps to `channel` with primary key `channel_id`.

A channel:

- belongs to one workspace;
- has many messages;
- has many users through `channel_user`;
- has many tasks;
- can be public or private through `is_private`.

Public channels are discoverable to workspace members and auto-join the authenticated user when they open the channel. Private channels require explicit membership in `channel_user`.

### Channel Messages

`App\Models\Message` maps to `message` with primary key `message_id`.

A message:

- belongs to a channel;
- belongs to a sender user;
- can reply to another message through `parent_id`;
- can have replies;
- can have polymorphic file attachments;
- can have polymorphic pins.

Top-level messages have `parent_id = null`; replies point at a parent message.

### Direct Messages

Direct messaging uses three main tables/models:

- `DmConversation`: one conversation container.
- `DmParticipant`: users included in a conversation.
- `DirectMessage`: individual messages in a conversation.

The current implementation starts or reuses a two-person conversation by finding a `dm_conversation` that has exactly the authenticated user and target user as participants.

### Tasks

`App\Models\Task` belongs to a channel and stores:

- creator user;
- optional assignee user;
- title and description;
- status from `App\Enums\TaskStatus`;
- optional due date.

Tasks are displayed in a Livewire board grouped by status: `pending`, `in_progress`, and `done`.

### Files

`App\Models\File` uses a polymorphic `attachable` relationship. Files can attach to:

- channel messages;
- direct messages.

The file table stores original name, path, size, and MIME type. Physical file bytes are stored on Laravel's `public` filesystem disk.

### Pinned Messages

`App\Models\PinnedMessage` is also polymorphic. In current UI usage, pins are attached to channel `Message` records.

The pinned-message panel queries pins whose attached message belongs to the active channel.

### Notifications

`App\Models\Notification` stores user-facing events such as:

- mentions;
- workspace join requests;
- workspace invites;
- accepted/rejected requests;
- accepted/rejected invites.

Notifications are queried over JSON routes for counts and recent notification lists. Some notification types are also broadcast in real time.

### Read States

Unread counts are not stored directly. They are calculated using read-state tables:

- `channel_read_state` stores a user's last read message per channel.
- `dm_read_state` stores a user's last read message per DM conversation.

The sidebars compare message IDs against the user's last-read message ID to calculate unread counts.

## Request and Page Flow

### Browser Entry

1. The browser requests a URL.
2. The web server sends it to `public/index.php`.
3. Laravel boots through `bootstrap/app.php`.
4. Routes from `routes/web.php` and `routes/auth.php` are matched.
5. Auth-protected routes require a logged-in user.
6. Controllers load models, authorize actions through policies, and return Blade views or redirects.
7. Blade views render layouts and Livewire components.
8. Livewire components perform dynamic reads/writes after the page loads.

### Root URL

`GET /` checks authentication:

- logged-in users are redirected to `workspaces.index`;
- guests are redirected to `login`.

### Dashboard

`GET /dashboard` also redirects to `workspaces.index`.

## Workspace Data Flow

### Listing Workspaces

Route: `GET /workspaces`

Controller: `WorkspaceController@index`

Flow:

1. The authenticated user's `user_id` is read.
2. The user's workspaces are loaded through the `User::workspaces()` relationship.
3. Other workspaces are loaded by excluding workspaces where this user already has a `workspace_members` row.
4. Pending join requests for the current user are loaded from `workspace_join_requests`.
5. `resources/views/workspaces/index.blade.php` receives:
   - `myWorkspaces`;
   - `otherWorkspaces`;
   - `pendingRequestIds`.

### Creating a Workspace

Route: `POST /workspaces`

Controller: `WorkspaceController@store`

Flow:

1. Request validates name, description, and optional avatar image.
2. If an avatar is provided, `CloudinaryImageService` uploads it when configured; otherwise Laravel stores it on the public disk.
3. A `workspace` row is created.
4. A `workspace_members` row is created for the creator with admin role.
5. User is redirected to the workspace show page.

### Viewing a Workspace

Route: `GET /workspaces/{workspace}`

Controller: `WorkspaceController@show`

Flow:

1. The current user's membership is checked in `workspace_members`.
2. Admin status is derived from the member role.
3. Visible channels are loaded:
   - all public channels;
   - private channels where the user exists in `channel_user`.
4. Workspace members are loaded with user records.
5. Pending join requests are loaded for admins.
6. The current user's own join request is loaded.
7. The workspace show Blade page renders workspace details, channels, members, and admin controls.

### Requesting to Join a Workspace

Route: `POST /workspaces/{workspace}/join`

Controller: `WorkspaceController@join`

Flow:

1. If the user is already a member, the request is rejected with an info message.
2. A `workspace_join_requests` row is created or updated to `pending`.
3. Workspace admins are found in `workspace_members`.
4. A `notifications` row is created for each admin.
5. User is redirected back to workspace listing.

### Approving or Rejecting Join Requests

Routes:

- `POST /workspaces/{workspace}/join-requests/{joinRequest}/approve`
- `POST /workspaces/{workspace}/join-requests/{joinRequest}/reject`

Controller:

- `WorkspaceController@approveJoin`
- `WorkspaceController@rejectJoin`

Flow for approval:

1. Admin authorization runs through the workspace policy.
2. Join request status becomes `accepted`.
3. A `workspace_members` row is created for the requester with member role.
4. A notification is created for the requester.

Flow for rejection:

1. Admin authorization runs.
2. Join request status becomes `rejected`.
3. A notification is created for the requester.

### Inviting a User

Route: `POST /workspaces/{workspace}/invite`

Controller: `WorkspaceController@invite`

Flow:

1. Admin authorization runs.
2. The submitted email must exist in `users`.
3. Existing membership and pending request are checked.
4. A pending `workspace_join_requests` row is created.
5. A `workspace_invite` notification is created for the invitee.

### Accepting or Rejecting an Invite

Routes:

- `POST /workspaces/{workspace}/invite/accept`
- `POST /workspaces/{workspace}/invite/reject`

Flow for accept:

1. The user's pending join request is loaded.
2. Request status becomes `accepted`.
3. The user is added to `workspace_members`.
4. A notification is sent to an admin.

Flow for reject:

1. The user's pending join request is loaded.
2. Request status becomes `rejected`.
3. Admins are notified.

## Channel Data Flow

### Creating a Channel

Route: `POST /workspaces/{workspace}/channels`

Controller: `ChannelController@store`

Flow:

1. Workspace update authorization runs.
2. Request validates channel name, privacy flag, and optional member IDs.
3. A `channel` row is created.
4. The creator is inserted into `channel_user`.
5. If the channel is private, selected members are inserted into `channel_user`.
6. User is redirected to the channel show page.

### Viewing a Channel

Route: `GET /channels/{channel}`

Controller: `ChannelController@show`

Flow:

1. Channel view authorization runs.
2. The channel's workspace is loaded.
3. If the channel is public, the authenticated user is added to `channel_user`.
4. `resources/views/channels/show.blade.php` renders the channel page.
5. Livewire components on the page load messages, channel sidebar, tasks, and pins.

### Joining or Leaving a Channel

Routes:

- `POST /workspaces/{workspace}/channels/{channel}/join`
- `POST /workspaces/{workspace}/channels/{channel}/leave`

Flow:

- Join inserts or reuses a `channel_user` row.
- Leave deletes the user's `channel_user` row.

## Channel Chat Data Flow

Primary component: `App\Livewire\ChatWindow`

View: `resources/views/livewire/chat-window.blade.php`

### Loading Chat

When mounted:

1. The component receives a `Channel`.
2. It queries the latest 50 top-level `message` rows for the channel.
3. It eager-loads sender, replies, reply senders, files, and pins.
4. It reverses results into chronological display order.
5. It stores the latest message ID.
6. It writes/updates `channel_read_state` for the current user.

### Sending a Message

Livewire action: `ChatWindow::send`

Flow:

1. Authorization checks whether the user can send a message in the channel.
2. Request state validates:
   - body required unless attachment exists;
   - body max 4000 chars;
   - optional attachment max 10 MB.
3. A `message` row is created with:
   - `channel_id`;
   - `sender_id`;
   - optional `parent_id`;
   - `msg_body`;
   - `msg_type` of `text` or `file`;
   - `sent_at`.
4. If an attachment exists:
   - the file is stored under `attachments/{message_id}` on the public disk;
   - a polymorphic `file` row is created for the message.
5. The message body is scanned for `@username` mentions.
6. Mentioned users get `notifications` rows.
7. The message loads sender/files/pins for immediate UI insertion.
8. A `MessageSent` event is broadcast on termination.
9. Mention notifications are broadcast through `NotificationCreated`.
10. The local Livewire message list is appended.
11. The current user's channel read state is updated.
12. The component dispatches a browser/Livewire `message-sent` event.

### Receiving Messages in Real Time

Broadcast authorization: `routes/channels.php`

Private channel: `channel.{channelId}`

Authorization rule:

- the user must have a `channel_user` row for that channel.

When a broadcast arrives, the Livewire listener appends the message, updates `latestMessageId`, marks the channel as read, and scrolls the UI.

The component also polls with `refreshMessages`, so the UI can recover from missed websocket events or disabled broadcasting.

### Typing Indicator

Flow:

1. `broadcastTyping` stores a short-lived cache key:
   - `typing-channel-{channel_id}-{user_id}`
2. It broadcasts `UserTyping` to others.
3. `checkTyping` scans workspace member IDs and reads matching cache keys.
4. The component displays typing names when present.

## Direct Message Data Flow

Primary component: `App\Livewire\DirectMessageWindow`

### Starting a DM

Route: `POST /dms/start/{user}`

Controller: `DirectMessageController@start`

Flow:

1. The authenticated user's ID and target user's ID are read.
2. The app searches for a conversation that has both users as participants and exactly two participants.
3. If none exists:
   - a `dm_conversation` row is created;
   - two `dm_participant` rows are created.
4. User is redirected to the DM show page.

### Viewing DMs

Route: `GET /dms`

Controller: `DirectMessageController@index`

Flow:

1. Conversations where the current user is a participant are loaded.
2. Participants and the latest direct message are eager-loaded.
3. `resources/views/dms/index.blade.php` renders the list.

Route: `GET /dms/{conversation}`

Controller: `DirectMessageController@show`

Flow:

1. Conversation view authorization runs.
2. Participants and users are loaded.
3. `resources/views/dms/show.blade.php` renders the DM page.
4. `DirectMessageWindow` loads the messages.

### Sending a DM

Livewire action: `DirectMessageWindow::send`

Flow is parallel to channel chat:

1. Authorize the user can view/send in the conversation.
2. Validate message body and optional attachment.
3. Create a `direct_message` row.
4. Store any attachment and create a polymorphic `file` row.
5. Parse mentions and create notifications.
6. Broadcast `DirectMessageSent` on private channel `dm.{conversationId}`.
7. Broadcast mention notifications.
8. Append message locally.
9. Update `dm_read_state`.

### DM Broadcast Authorization

Private channel: `dm.{conversationId}`

Authorization rule:

- the user must have a `dm_participant` row for that conversation.

## Sidebar and Unread Count Flow

### Channel Sidebar

Component: `App\Livewire\ChannelSidebar`

Flow:

1. Loads channels in the active workspace where the user is present in `channel_user`.
2. Collects channel IDs.
3. Left-joins `message` to `channel_read_state`.
4. Counts messages with no read state or with `message_id > last_read_message_id`.
5. Maps each channel to display data:
   - ID;
   - name;
   - privacy;
   - unread count;
   - URL.

### DM Sidebar

Component: `App\Livewire\DmSidebar`

Flow:

1. Loads conversations where the user is a participant.
2. Loads participants and their users.
3. Left-joins `direct_message` to `dm_read_state`.
4. Counts unread messages with no read state or with `dm_message_id > last_read_message_id`.
5. Maps each conversation to display data:
   - conversation ID;
   - other user's username/avatar;
   - online status from cache;
   - unread count;
   - URL.

## Task Data Flow

### Creating, Editing, and Deleting Tasks

Routes are nested under channels:

- `GET /channels/{channel}/tasks/create`
- `POST /channels/{channel}/tasks`
- `GET /channels/{channel}/tasks/{task}/edit`
- `PATCH /channels/{channel}/tasks/{task}`
- `DELETE /channels/{channel}/tasks/{task}`

Controller: `TaskController`

Flow:

1. Sending/updating/deleting authorization runs through channel/task policies.
2. Request validates title, description, assignee, due date, and status.
3. Task records are created, updated, or deleted.
4. User returns to the channel page.

### Live Task Board

Component: `App\Livewire\TaskBoard`

Flow:

1. On mount, it loads all tasks for the channel with creator and assignee.
2. It groups tasks into columns keyed by task status.
3. When a task status changes, it:
   - authorizes the update;
   - updates the task status;
   - broadcasts `TaskUpdated` to others on the private channel;
   - reloads the task columns.
4. Other connected users receive `TaskUpdated` and reload the board.

## Pinned Message Flow

### Pinning from Chat

Livewire action: `ChatWindow::pinMessage`

Flow:

1. User must be authorized to send messages in the channel.
2. The component checks if a `pinned_message` row already exists for the message.
3. If it exists, it deletes the pin.
4. If it does not exist, it creates the pin with `pinned_by` set to the current user.
5. Messages are reloaded so pin state refreshes.

### Viewing Pins

Component: `App\Livewire\PinnedMessagesPanel`

Flow:

1. Counts pins where `pinnable_type` is `Message` and the attached message belongs to the active channel.
2. When opened, loads pinned messages with sender and pinning user.
3. Allows authorized users to unpin.

## File Attachment Flow

There are two attachment paths in the code:

1. Inline attachments from `ChatWindow` and `DirectMessageWindow`.
2. Generic upload/download routes in `FileController`.

### Inline Attachments

Channel message:

1. User sends message with attachment.
2. Livewire stores the file under `attachments/{message_id}`.
3. A `file` row is created with `attachable_type = App\Models\Message`.

Direct message:

1. User sends DM with attachment.
2. Livewire stores the file under `attachments/dm-{dm_message_id}`.
3. A `file` row is created with `attachable_type = App\Models\DirectMessage`.

### Downloading Files

Route: `GET /files/{file}/download`

Controller: `FileController@download`

Flow:

1. The file's polymorphic parent is loaded.
2. Authorization is based on parent type:
   - channel message: authorize viewing the message's channel;
   - direct message: authorize viewing the message's conversation.
3. The file path must exist on the public disk.
4. Laravel streams the file back with content type and length headers.

## Notification Flow

Notifications are stored in the `notifications` table and exposed through JSON endpoints.

### Creation Points

Notifications are created when:

- a user mentions another user in a channel message;
- a user mentions another user in a DM;
- a user requests to join a workspace;
- an admin approves/rejects a join request;
- an admin invites a user to a workspace;
- an invitee accepts/rejects an invite.

### API Endpoints

- `GET /notifications/count` returns the current user's unseen count.
- `GET /notifications` returns the latest 30 notifications with sender/workspace.
- `POST /notifications/mark-seen` marks unseen notifications as seen.

### Real-Time Notifications

Mention notifications created from chat/DM are broadcast with `NotificationCreated`.

Private channel: `user.{id}`

Authorization rule:

- the authenticated user's `user_id` must match the channel ID.

## Real-Time Broadcasting Flow

Broadcasting uses Laravel Echo on the frontend and Laravel Reverb/Pusher-compatible broadcasting on the backend.

Frontend setup:

- `resources/js/echo.js` reads Reverb config from Vite environment variables or meta tags.
- It creates `window.Echo` using the `reverb` broadcaster.

Backend setup:

- `routes/channels.php` authorizes private channels.
- Events in `app/Events/` describe payloads and channels.
- Livewire components listen to Echo events with `#[On('echo-private:...')]` attributes.

Private broadcast channels:

- `App.Models.User.{id}`: user model private channel.
- `user.{id}`: user notifications.
- `workspace.{workspaceId}`: workspace-level events for members.
- `channel.{channelId}`: channel messages, typing, and task updates.
- `dm.{conversationId}`: direct messages and DM typing.

## Online Presence Flow

Middleware: `App\Http\Middleware\UpdateUserLastSeen`

Flow:

1. On each authenticated web request, the middleware reads the current `user_id`.
2. It throttles writes with cache key `user-online-throttle-{user_id}` for 30 seconds.
3. If allowed, it stores `user-online-{user_id}` for 5 minutes.
4. `User::isOnline()` checks whether `user-online-{user_id}` exists in cache.
5. DM sidebar uses this to show whether the other participant appears online.

This is request-driven presence, not websocket-presence-channel presence.

## Authorization Flow

Authorization is handled with policies in `app/Policies/`.

Important checks appear in controllers and components:

- Workspace update/delete/member management uses `WorkspacePolicy`.
- Channel view/update/delete/send-message checks use `ChannelPolicy`.
- DM conversation access uses `DirectMessagePolicy` or related policy logic.
- Task update/delete uses `TaskPolicy`.
- Message and pin actions use channel/message authorization.

Broadcast authorization is separate and lives in `routes/channels.php`. It controls who may subscribe to private real-time channels.

## Frontend Rendering Flow

The UI is a mix of Blade and Livewire:

1. Controllers return full Blade pages under `resources/views/...`.
2. Shared page chrome lives in `resources/views/layouts/`.
3. Reusable Blade components live in `resources/views/components/`.
4. Interactive areas are Livewire components:
   - `chat-window`;
   - `direct-message-window`;
   - `channel-sidebar`;
   - `dm-sidebar`;
   - `task-board`;
   - `pinned-messages-panel`.
5. CSS comes from `resources/css/app.css` and `resources/css/theme.css`.
6. JavaScript is bundled by Vite from `resources/js/app.js`.

## High-Level Data-Flow Summary

```text
Browser
  -> public/index.php
  -> Laravel bootstrap/app.php
  -> routes/web.php or routes/auth.php
  -> Controller or Livewire action
  -> Policy authorization
  -> Eloquent model query/update
  -> Database tables
  -> Blade/Livewire response
  -> Browser UI
```

For real-time paths:

```text
Livewire action
  -> Eloquent write
  -> Event broadcast
  -> routes/channels.php authorizes subscribers
  -> Reverb/Echo websocket delivery
  -> Livewire #[On(...)] listener
  -> Component state refresh
  -> Browser UI updates
```

For file paths:

```text
Livewire/controller upload
  -> Laravel validation
  -> storage/app/public/attachments/...
  -> file table metadata
  -> message/direct-message polymorphic relationship
  -> authorized download route
  -> streamed response
```

For unread counts:

```text
Message inserted
  -> user opens channel/DM
  -> read-state row stores latest message ID
  -> sidebar left-joins messages against read state
  -> count where message ID is newer than last read
  -> unread badge in UI
```

## Main User Journeys

### User logs in and enters the portal

1. Guest hits `/`.
2. App redirects to `/login`.
3. Auth controller validates credentials against `users.password_hash`.
4. Logged-in user redirects to workspace listing.

### User creates a workspace

1. User submits workspace form.
2. Workspace row is created.
3. Creator is added as admin in `workspace_members`.
4. User lands on the workspace page.

### User joins a workspace

1. User clicks join on a non-member workspace.
2. App creates a pending join request.
3. Admins receive notifications.
4. Admin approves.
5. User is inserted into `workspace_members`.
6. User receives acceptance notification.

### Admin creates a private channel

1. Admin opens create-channel form inside a workspace.
2. Admin selects private mode and member IDs.
3. Channel row is created.
4. Creator and selected members are inserted into `channel_user`.
5. Only those users can view and subscribe to the channel.

### User sends a channel message with a mention

1. User types message in `ChatWindow`.
2. Livewire creates `message`.
3. If text contains `@username`, notifications are created.
4. Message is broadcast to `channel.{channelId}`.
5. Mention notifications are broadcast to `user.{id}`.
6. Other users' chat windows update in real time.
7. Sidebar unread counts update based on read-state comparison.

### User sends a direct message

1. User starts or opens a DM conversation.
2. Livewire creates `direct_message`.
3. Optional file is stored and linked.
4. Message is broadcast to `dm.{conversationId}`.
5. Participants receive the update if authorized for that private channel.

### User moves a task on the board

1. User changes task status in `TaskBoard`.
2. Livewire authorizes the task update.
3. Task status is updated.
4. `TaskUpdated` is broadcast.
5. Other users in the channel reload task columns.

## Notes and Observations

- The project uses singular table names for several domain tables, such as `workspace`, `channel`, `message`, `file`, and `pinned_message`.
- Primary keys are custom names such as `user_id`, `workspace_id`, `channel_id`, and `message_id`.
- Real-time features are designed to degrade gracefully because chat components also poll/refresh.
- Online presence is cache/request based, not a true websocket presence channel.
- Files are authorized through their parent message/conversation before download.
- Workspace invites and join requests share the `workspace_join_requests` table and are distinguished mainly by the surrounding controller flow and notification type.
- The generic `FileController@store` expects an existing message ID, while the main chat/DM upload flow creates files inline during message send.
